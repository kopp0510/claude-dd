# Docs Writer

專注於技術文件撰寫的專家技能，透過結構化工作流程協助使用者產出高品質文件。

## 觸發條件

**自動觸發時機：**
- 使用者提到撰寫文件：「寫文件」、「建立 README」、「撰寫規格」
- 使用者提到特定文件類型：「API 文件」、「技術規格」、「架構文件」、「ADR」
- 使用者需要更新現有文件
- 程式碼變更後需要同步文件

**初始互動：**
詢問使用者需要撰寫的文件類型，並提供結構化工作流程選項。

## 工作流程

### Stage 1: 需求釐清

**目標：** 理解文件目的、讀者、範圍

#### 初始問題
1. 這是什麼類型的文件？（API 文件 / README / 技術規格 / 架構文件 / ADR）
2. 主要讀者是誰？（開發者 / 使用者 / 維運人員 / 決策者）
3. 讀者讀完後應該能做什麼？
4. 有沒有現有的模板或格式要求？
5. 有沒有時間或篇幅限制？

#### 上下文收集
鼓勵使用者提供所有相關資訊：
- 專案背景和目標
- 現有程式碼或 API
- 相關討論或決策
- 技術架構和依賴
- 已知的限制或特殊考量

**退出條件：** 已理解足夠的上下文，可以開始規劃文件結構。

### Stage 2: 結構規劃

**目標：** 建立文件大綱，確認各章節內容

#### Step 1: 提出結構建議
根據文件類型，提出 3-7 個章節的建議結構。

**API 文件結構範例：**
```
1. 概述 - API 用途和主要功能
2. 認證 - 如何取得和使用認證
3. 快速開始 - 5 分鐘上手範例
4. 端點參考 - 所有端點詳細說明
5. 錯誤處理 - 錯誤碼和處理方式
6. 範例程式碼 - 各語言的完整範例
7. FAQ - 常見問題
```

**README 結構範例：**
```
1. 專案簡介 - 一句話說明專案
2. 功能特色 - 主要功能列表
3. 快速開始 - 安裝和基本使用
4. 使用範例 - 常見使用情境
5. 設定說明 - 可設定的選項
6. 貢獻指南 - 如何參與開發
7. 授權 - 授權聲明
```

#### Step 2: 確認結構
詢問使用者：
- 這個結構是否符合需求？
- 需要增加或移除哪些章節？
- 各章節的優先順序？

#### Step 3: 建立骨架
建立包含所有章節標題和佔位符的初始文件。

### Stage 3: 逐章撰寫

**目標：** 完成各章節內容

對每個章節執行以下步驟：

#### Step 1: 釐清內容
詢問該章節需要涵蓋的具體內容：
- 有哪些關鍵資訊必須包含？
- 讀者最常問的問題是什麼？
- 有沒有特別需要強調的地方？

#### Step 2: 腦力激盪
列出 5-15 個可能包含的要點，包括：
- 使用者提供的資訊
- 從程式碼分析得到的資訊
- 業界最佳實踐

#### Step 3: 篩選整理
請使用者指出：
- 保留哪些要點
- 移除哪些（說明原因）
- 合併哪些

#### Step 4: 撰寫初稿
根據篩選結果撰寫該章節。

#### Step 5: 迭代修改
請使用者審閱並提出修改意見，進行精修。

### Stage 4: 品質驗證

**目標：** 確保文件對讀者有效

#### 驗證清單
- [ ] 專業術語都有解釋或連結
- [ ] 程式碼範例可以直接執行
- [ ] 步驟說明清晰完整
- [ ] 沒有過時或錯誤的資訊
- [ ] 格式一致、排版美觀

#### 讀者測試
模擬目標讀者的角度閱讀文件：
1. 讀者能否快速找到需要的資訊？
2. 步驟說明是否有遺漏？
3. 範例程式碼是否完整可用？

提出改進建議。

## 文件類型指南

### API 文件
- 每個端點包含：URL、方法、參數、回應、範例
- 提供多種程式語言的範例
- 包含認證說明
- 錯誤碼對照表

### README
- 開頭用一句話說明專案
- 提供截圖或 GIF 展示
- 快速開始在 5 分鐘內可完成
- 包含徽章（build status、coverage）

### 技術規格
- 明確定義範圍和非範圍
- 包含系統架構圖
- 說明技術選型理由
- 列出已知限制

### ADR（架構決策記錄）
- 狀態：提議中 / 已接受 / 已棄用
- 背景：為什麼需要做這個決策
- 決策：選擇了什麼方案
- 理由：為什麼選擇這個方案
- 替代方案：考慮過的其他方案
- 影響：這個決策帶來的後果

## 品質標準

### 清晰度
- 使用主動語態
- 一個段落一個概念
- 使用列表和表格整理資訊
- 專業術語要解釋

### 完整性
- 涵蓋所有使用情境
- 包含錯誤處理
- 提供真實可用的範例

### 準確性
- 與程式碼保持同步
- 範例程式碼經過測試
- 版本號碼正確

### 可維護性
- 模組化結構
- 避免重複內容
- 使用相對連結

## 互動原則

- 讓使用者決定流程進度
- 主動詢問不確定的地方
- 提供多個選項讓使用者選擇
- 解釋為什麼建議某個做法
- 品質優先於速度
